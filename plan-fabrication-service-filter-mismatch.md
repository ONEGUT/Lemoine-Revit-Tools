# Plan — Fix Fabrication Service discover/filter value mismatch

## Problem

Filters auto-generated by the **Discover Rules** tool for the **Fabrication Service**
parameter never match any elements, so the view override (colour) is never applied —
even though the rule looks correct in the filter editor.

### Root cause

The value is **read** during the scan through a different parameter than the one the
filter later **compares against**:

- **Scan (read):** `DiscoverEventHandler.ReadParameterValue` (`DiscoverEventHandler.cs:285`)
  calls `el.LookupParameter("Fabrication Service")`, which resolves to the property-palette
  field. Its value is the **composite** string `DVG - MP: Chilled Water Supply`
  (`<abbreviation>: <service name>`). That whole string is stored as the rule's `Match`
  keyword (`DiscoverEventHandler.cs:364`).

- **Apply (match):** `AutoFiltersEventHandler` maps the display name to a built-in
  (`AutoFiltersEventHandler.cs:118`):
  `["Fabrication Service"] = BuiltInParameter.FABRICATION_SERVICE_NAME`.
  In the filter editor this is the **"Fabrication Service Name"** parameter, whose value is
  only the service-name portion (`Chilled Water Supply`) — no `DVG - MP:` prefix.

The generated rule is therefore `FABRICATION_SERVICE_NAME contains "DVG - MP: Chilled Water Supply"`,
which can never be satisfied because the parameter value is just `Chilled Water Supply`.

The read side and the match side were written against two different notions of
"Fabrication Service" and drifted apart. (Note: a correct, `ServiceName`-property-based reader
already exists at `AutoFiltersEventHandler.cs:845` but is not used by the scan.)

## Fix

Make the scan capture the value **the filter parameter actually compares against**, so the
`contains` keyword lines up.

### File changed
- `Source/Tools/T01-AutoFilters/DiscoverEventHandler.cs`

### Change
In `ReadParameterValue`, special-case `"Fabrication Service"` to read it through
`BuiltInParameter.FABRICATION_SERVICE_NAME` using `AsString()` — the same parameter and value
basis the generated filter uses — instead of `LookupParameter("Fabrication Service")`.

```csharp
if (paramName == "Fabrication Service")
{
    // Read the SAME built-in the filter binds (FABRICATION_SERVICE_NAME), so the
    // captured keyword matches what the view filter compares against. LookupParameter
    // by display name returns the palette composite ("DVG - MP: Chilled Water Supply"),
    // which the FABRICATION_SERVICE_NAME filter — value "Chilled Water Supply" — never matches.
    var fp = el.get_Parameter(BuiltInParameter.FABRICATION_SERVICE_NAME);
    if (fp != null && fp.HasValue)
    {
        string? fv = fp.AsString();
        if (!string.IsNullOrWhiteSpace(fv)) return fv;
        fv = fp.AsValueString();
        if (!string.IsNullOrWhiteSpace(fv)) return fv;
    }
    return null;
}
```

This is a read-only, surgical change to the scan path. It does not alter how filters are
built or applied; it only ensures the discovered keyword is the service-name string the
filter checks.

### Out of scope (noting, not changing now)
- The bipMap label `"Fabrication Service"` → `FABRICATION_SERVICE_NAME` stays as-is (the
  filter side is correct; the scan side was wrong).
- No change to `ClashEngine.cs:59` (same mapping, but its read path differs and is not
  reported broken).

Already-generated filters keep the bad `DVG - MP: ...` keyword. Either re-run Discover after
the fix, or manually edit the existing filter value to drop the prefix. (Will mention in the
